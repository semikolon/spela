# spela — AI-Agent-Ready Media Controller

## NO PERSONAL DATA (MANDATORY)
**Public repo.** Never commit real names, IPs, API keys, device names, or household details to code/docs. Use placeholders. Runtime config (`config.toml`, `state.json`) is user-local and gitignored.

## Overview

Rust CLI + HTTP API server for torrent-to-Chromecast streaming. Single 5.2MB binary, zero runtime deps on target (except webtorrent-cli and ffmpeg).

**Status**: v3.1.0 shipping on Darwin (Apr 19, 2026). HLS cast pipeline, Smart Resume, self-healing disk cache, **5-tier Sarpetorp-tuned ranker** (1080p target, 2160p demoted below 480p, DV hard gate), cast_health_monitor, DNS DNAT hijack for Chromecast LAN hostname resolution. Seeking via server-side restart works (native seeking via Custom Receiver App still blocked on $5 registration).

**Architecture**: `spela server` on Darwin (`darwin.home:7890`). `spela <command>` thin HTTP client from any LAN machine. Voice assistants call HTTP API directly.

## Usage

```bash
spela search "Good Luck Have Fun Dont Die"       # Auto-detects movie vs TV
spela search "legion" --season 1 --episode 5   # Search + rank results
spela play 1                                     # Play result #1 (auto: magnet, file_index, metadata)
spela play 1 --cast "Vardagsrum"                 # Play to living room TV
spela play 1 --no-intro                          # Skip intro bumper
spela next                                       # Next episode
spela pause / resume / stop                      # Playback control
spela status / history / targets                 # Info
```

`spela play <N>` is the primary flow. Carries all metadata (file_index, IMDB, show, season, episode) automatically from the last search.

## Key Files

- `src/main.rs` — CLI (clap) + server startup. Detects `play 1` (result ID) vs `play magnet:...`
- `src/server.rs` — axum HTTP API, 21 endpoints (including Custom Receiver: cast-config, seek-restart, position, retry). Orchestrates search→play→cast pipeline
- `src/search.rs` — TMDB + Torrentio. **Auto-detect movie vs TV** via TMDB multi-search. **5-tier Sarpetorp-tuned ranker** (v3.1.0, shared `pub fn rank_results_mut`): (1) single-file > pack, (2) non-DV > DV (HARD gate — GTX 1650 NVENC can't parse DV profile 5/7 RPU), (3) target resolution `1080p > 720p > 480p > 2160p > unknown` with ≥50-seed viability (2160p dead-last because TVs max at 1080p native + 4K NVENC transcode is ~3x heavier), (4) H.264 > HEVC within same resolution (insta-play tiebreak, ≥5-seed threshold), (5) more seeds. The v3.0.0 → v3.1.0 flip: resolution tier was promoted ABOVE codec preference because HEVC→H.264 transcode at 1080p runs ~6x realtime, fast enough to absorb the cold-start cost — so 1080p HEVC now beats 720p H.264 on a 1080p TV
- `src/cast.rs` — Native Chromecast via rust_cast + mdns-sd. 3x retry + IP cache + known device fallback
- `src/torrent_engine.rs` — pure-Rust torrent engine wrapping `librqbit::Session` (since v3.3.0 / Apr 30, 2026). spela-shaped surface: `start` / `progress` / `handle` / `stop`, `validate_magnet_uri` (SSRF defense), `shift_librqbit_id` / `unshift_librqbit_id` (preserves the `pid==0 = Local Bypass` sentinel against librqbit's TorrentId-starts-at-0). Opens `Session::new_with_opts` with peer-connection timeouts (15s/60s/120s) + concurrent_init_limit=4
- `src/torrent_stream.rs` — axum HTTP Range handler that pipes `librqbit::FileStream` (`AsyncRead + AsyncSeek`) into `axum::body::Body` via `tokio_util::io::ReaderStream`. Pure `parse_range_header` covers all RFC 7233 forms (open-ended, bounded, suffix, multi-range, malformed). The `/torrent/{id}/stream/{file_idx}` route is **loopback-only** via the `require_loopback_source` middleware (H4)
- `src/torrent.rs` — slimmed to ffmpeg-PID utilities + a one-shot post-upgrade webtorrent-orphan sweep at startup. Pre-v3.3.0 this wrapped the Node `webtorrent-cli` subprocess
- `src/transcode.rs` — ffprobe codec+duration detection, ffmpeg audio transcode (EAC3/DTS/AC3→AAC) + video transcode (HEVC→H.264 via NVENC) + intro concat
- `src/subtitles.rs` — Stremio OpenSubtitles v3 (zero auth), SRT→WebVTT
- `src/disk.rs` — two-layer disk safety: 10 GB internal media cap (sparse-aware via `metadata.blocks() * 512`) + 20 GB host-filesystem free-space floor (via `df -Pk`) + age-based prune (top-level files AND directories) + **LRU pressure eviction via `prune_to_fit`** so the cap is a self-maintaining upper bound instead of a refusal wall (Apr 15, 2026). `title_matches_active` tokenizes filename vs active title so `The.Boys.S05E03.FLUX.mkv` correctly matches `The Boys S05E03`. See `OPERATIONS.md` for the full defense-in-depth map
- `src/state.rs` — state.json + last_search.json (play-by-id) + resume_positions (per-episode keyed for TV shows via `resume_position_key` + `extract_se_suffix` — Apr 15, 2026 fix for IMDb-ID collision across episodes of the same show)
- `static/cast-receiver.html` — Custom Cast Receiver (Shaka Player + CAF v3, ~200 lines)
- `src/config.rs` — `~/.config/spela/config.toml`. **Hardcoded XDG path on every platform** because Rust's `dirs::config_dir()` returns `~/Library/Application Support` on macOS, which silently hid the Mac CLI's user config and dropped it back to `localhost:7890`
- `OPERATIONS.md` — worker safety plan, stale WebTorrent prevention, emergency cleanup rules

## Build & Deploy

cd ~/Projects/spela && git push   # from Mac
ssh darwin 'cd ~/Projects/spela && git pull && cargo build --release'
ssh darwin 'sudo systemctl restart spela'
```

`~/.local/bin/spela` on Darwin is a symlink to `~/Projects/spela/target/release/spela` (verified Apr 30, 2026 — `lrwxrwxrwx ... -> /home/fredrik/Projects/spela/target/release/spela`). `cargo build --release` rewrites the inode the symlink points at; the running spela process keeps the OLD inode open until `systemctl restart` re-execs against the symlink. **No `cp` step needed** — earlier versions of this doc had `cp ~/Projects/spela/target/release/spela ~/.local/bin/` which errors with "are the same file" (Apr 30 deploy hit this).

## Hard-Won Lessons

- **v3.3.0 — librqbit migration + Apr 30 security audit hardening** (Apr 29-30, 2026): replaced webtorrent-cli with embedded librqbit; 13 audit-driven commits across H1-H4 + tier 3 + RGR refactors; +79 tests. Full session report: [`docs/session_2026_04_29_30_librqbit_migration_and_security_audit.md`](docs/session_2026_04_29_30_librqbit_migration_and_security_audit.md). Quick gotchas-list (one-line each, ordered by future-session likelihood-of-encountering):
  - **rustls 0.23 needs `CryptoProvider::install_default()` early in `main`** — without it, librqbit's first HTTPS tracker fetch panics; the panic poisoned the cast `Mutex` and cascaded `PoisonError` across every axum task. Pinned `aws-lc-rs` provider + `install_default().ok()` in `main.rs`. Commit `8d35a78`.
  - **librqbit allocates `TorrentId = 0` for first torrent — collides with spela's `pid==0 = Local Bypass` sentinel.** Engine wrapper applies `+1` shift (`shift_librqbit_id` / `unshift_librqbit_id`). Commit `6c040de`.
  - **`/torrent/{id}/stream/{file_idx}` is loopback-only** — the URL builder hardcodes `127.0.0.1` (not `stream_host`) so ffmpeg's source IP is loopback and passes the `require_loopback_source` middleware. Chromecast still uses `stream_host` for `/hls/*`. Commit `610f7a8`.
  - **Reject non-magnet URIs at the HTTP boundary** — `librqbit::AddTorrent::Url` accepts http(s):// and fetches them, which would have been an SSRF pivot. `validate_magnet_uri()` defends. Universal pattern in global CLAUDE.md § "Code Changes". Commit `7445530`.
  - **Use `lock_recover()` not `.lock().unwrap()`** — poison cascade is a real risk in long-running async servers. Universal pattern in global CLAUDE.md § "Code Changes". Commit `61d18e2`.
  - **Host-header allowlist middleware** + tightened CORS — DNS-rebinding defense. Commit `14671d4`.
  - **Local Bypass decision extracted into `find_local_bypass_match`** (RGR + 8 tests, Apr 8/15/18/19/25/28/29 incident-cluster pinned). Commit `552b49e`.
  - **cast_health_monitor sub-decision RGR** — `evaluate_buffering_state` / `is_natural_eof` / `should_save_position` extracted; inline chains became enum matches. Apr 18/19/29 incident pins. Full state-machine extraction deferred per user's *"have good judgment"* scope-calibration. Commit `05631cc`.
- **English audio track preference** (Apr 18) — `detect_codecs` now returns `CodecInfo` with `audio_index` (preferred English track). ffmpeg uses `-map 0:a:{audio_index}` instead of `[0:a]` (which picks the DEFAULT, often Russian for dual-audio releases). Without this, Send.Help.2026 played in Russian.
- **CRLF subtitle parsing** (Apr 18) — OpenSubtitles SRT files use `\r\n` line endings. `shift_srt` used `split("\n\n")` which silently failed on `\r\n\r\n` separators → 0 entries retained → empty subtitle file → ffmpeg crash → blue cast icon. Fix: normalize `\r\n` → `\n` before parsing. Also: return `None` (skip subtitles) when 0 entries retained instead of passing an empty SRT to ffmpeg.
- **HLS pre-buffer 10 segments** (Apr 18) — old pre-buffer waited for 1 segment (6s content). Chromecast consumed segments at ~1x realtime and caught the transcode frontier after ~30s → spinner. Fix: wait for 10 segments (~60s head start). Timeout: 90s with intro, 60s without.
- **Intro concat breaks Chromecast HLS** (Apr 18, UNDER INVESTIGATION) — with `--no-intro`, movie plays smoothly past segment 5. WITH intro, Chromecast buffers indefinitely at ~segment 5. Workaround: `--no-intro`. Apr 19 session produced a specific hypothesis (NVENC scene-detection IDR at the intro→main seam desyncs GOP cadence from the HLS segmenter's 6s clock) + concrete proposed fix — see `TODO.md` § "Intro concat investigation".
- **cast_health_monitor must NOT kill on BUFFERING** (Apr 18) — BUFFERING is a transient state (Chromecast alive, waiting for data). Killing workers on BUFFERING converted a recoverable condition into a permanent stream death. Now logged for observability only. Only IDLE/UNKNOWN/error should trigger cleanup. See global CLAUDE.md § "Symptoms are signals" + `docs/NINE_PRINCIPLES_OF_SYSTEM_INTERVENTION.md`.
- **Percentage-based "end-of-episode" early-kill removed** (Apr 19) — `cast_health_monitor` used to tear down the stream once absolute position crossed 92% of duration, conflating a single constant with two distinct policies: (a) when to CLEAR the saved resume HWM (sensible — don't auto-resume from credits), and (b) when to KILL the cast session (not sensible). Same-constant coupling meant any film whose climax falls past 92% was uncompletable. Incident: Send Help 2026 (6776s, 113 min) killed at 1:43:54 with 8:42 of climax remaining, stream died mid-scene, user saw the blue cast icon. Fix: `state::HWM_CLEAR_FRACTION` raised to 0.96 and kept as the ONLY threshold for HWM clearing; `cast_health_monitor` no longer has a percentage-based cleanup branch at all. Real EOF now handled by (1) Chromecast reporting IDLE (player_state transitions to IDLE at EOF, cleanup fires after `IDLE_FAILURE_THRESHOLD` polls ≈15s) and (2) the Reaper's duration-aware grace period. Belt-and-suspenders: when IDLE-driven cleanup fires past `HWM_CLEAR_FRACTION`, the HWM is explicitly cleared (prevents stale HWM leaking into next TV episode via the same `imdb_id`). Regression pins: `test_save_position_smart_completion_threshold` (95.5% survives, 97.1% clears), `test_hwm_clear_fraction_invariant` (0.96 floor). **Generic lesson**: when one constant governs two independent policies, any threshold tuning for one policy silently distorts the other. Split them at the constant level even when the values happen to coincide today — the coincidence is load-bearing.

- **NVENC h264 cannot encode 10-bit input** (Apr 28-29) — Hijack S2E2 MeGusta release was HEVC Main 10 (`yuv420p10le`); ffmpeg's NVENC h264 encoder rejected it instantly: `[h264_nvenc] 10 bit encode not supported / No capable devices found / Nothing was written into output file`. Result: zero segments, receiver IDLE'd at `<init>` after 20s, looked like a "Chromecast wedge" until the ffmpeg.log was read. spela's ranker doesn't filter by bit depth; HEVC sources from groups like MeGusta and ELiTE often ship 10-bit. **Fix** (commit `ee38f92`): append `format=yuv420p` to every video filter chain feeding NVENC (six sites — three in `transcode()` legacy path + three in `transcode_hls()`: intro-concat filter_complex v0/v1, simple `-vf subtitles`, bare video-reencode-without-subs). No-op for already-8-bit input. Forces 8-bit downconvert before NVENC sees the frames. **Generic lesson**: encoder errors that produce zero output AND no obvious downstream failure indicator (cast still says "streaming" because ffmpeg launched, just produced empty output) are the worst class — read the ffmpeg.log FIRST when receiver IDLEs at <init> with low stream_age. The directive "Make failures visible before fixable" is exactly this — the symptom (cast IDLE) was a downstream signal, not the source.

- **Recast budget: rate-limited unbounded > lifetime cap-of-1** (Apr 29) — Apr 25's auto-recast capped at one attempt per stream lifecycle to prevent infinite LOAD/IDLE loops on truly wedged devices. Right answer for the failure class it was built for (single random IDLE, retry, give up). WRONG answer for the class hit Apr 28-29: receiver IDLEs every 15-30 minutes of sustained playback, recovers with each recast, IDLEs again. Each recast was successful recovery, not "burning cycles on a wedge" — but the lifetime cap rejected after the first. Hijack S2E2 incident: 6 rapid Playing↔Buffering oscillations at 38-40 min into stream, all rejected because the budget burned on the first IDLE. **Fix**: `should_attempt_recast` now takes `secs_since_last_recast: Option<u64>` (instead of `recast_attempts: u32`); guards are have_valid_hwm + stream_age_floor + cooldown. `RECAST_COOLDOWN_SECS = 90` (well above max recast→Playing recovery time ~25s, well below typical healthy-recast cycle ~3-15 min). The cap-of-1 conflated wedge-detection with attempt-limiting; rate-limit handles wedge protection cleanly without rejecting recoverable cycles. Companion: `MAX_BUFFERING_DURATION_SECS = 60` — Apr 18's "BUFFERING is transient, don't kill" had no time bound; receiver could stay in BUFFERING forever (Apr 28-29 incident: receiver was BUFFERING for 5+ minutes when user fell asleep, monitor logging "transient" forever). 60s threshold escalates stalled BUFFERING to the same recast/cleanup path IDLE uses. Regression pins: `test_recast_unbounded_with_cooldown_satisfied`, `test_recast_blocked_during_cooldown`, `test_recast_blocks_simulated_wedge` (40 ticks @ 15s wedge → ≤7 recasts), `test_apr29_hijack_s2e2_recurring_idle_recovery` (incident replay), `test_recast_cooldown_is_stable`, `test_max_buffering_duration_is_stable`. **Generic lesson**: when a guard's purpose is "don't waste effort on a broken thing," express it as a frequency check, not a lifetime check — frequency naturally scales with how recoverable the failure is. Lifetime caps are a special case of "frequency = 1 per ∞" which is too restrictive when the underlying behavior is recurring-but-recoverable.

- **Auto-recast on first mid-stream IDLE + rich `idle_reason` diagnostics** (Apr 25) — Vardagsrum's old CrKey 1.56 Chromecast repeatedly dropped its CAF session after 11-12 minutes of sustained 1080p BluRay H.264 (~7 Mbps) playback. Pattern: healthy, then a single sudden transition to `player_state=IDLE` with zero error surface. Workers stayed alive the whole time (ffmpeg at 5.62× speed, torrent 42% ahead of playback), so cleanup was losing a perfectly good stream to a purely receiver-side flake. Two coordinated fixes: (1) **`PlaybackInfo.idle_reason`** — surfaced `rust_cast::channels::media::IdleReason` (Cancelled/Interrupted/Finished/Error) through the cast layer so the monitor logs a legible cause on every IDLE instead of just `player_state=IDLE`. Paired with a new INFO-level state-transition log (`PLAYING → BUFFERING → IDLE`) so the journal tells a story instead of forcing post-mortem guesswork. (2) **Auto-recast on first IDLE batch** — when `consecutive_failures` hits `IDLE_FAILURE_THRESHOLD` (3), `cast_health_monitor` calls `attempt_cast_recast(state, cast_name, hwm, ss_offset, duration)` which re-issues LOAD on the Default Media Receiver and then `cast.seek(hwm - ss_offset)` within the HLS stream. Workers untouched. Three guards via `should_attempt_recast(recast_attempts, stream_age_secs, have_valid_hwm)`: (a) exactly one recast per stream lifecycle — a wedged device cannot trap the monitor in an infinite LOAD/IDLE loop (second IDLE batch falls through to the normal cleanup path), (b) stream must have played ≥ `MIN_STREAM_AGE_FOR_RECAST_SECS` (60 s) — startup LOAD failures don't recover from re-LOAD (they need stream_host / DNS hairpin / manifest fixes), (c) a usable HWM must exist to seek to. EOF-mimicking IDLEs (HWM past `HWM_CLEAR_FRACTION`) are explicitly excluded from recast so we don't loop the credits. **Pre-cleanup dump** added at ERROR level right before `do_cleanup` — records `consecutive_failures`, `recast_attempts`, `stream_age`, `ss_offset`, `last_known_absolute`, `last_saved_position`, `duration`, `prev_state` — zero-IO, replaces the old black-box `"chromecast media session DEAD"` line with a replayable timeline. Regression pins: `test_recast_normal_mid_stream_death`, `test_recast_caps_at_one_attempt_per_stream` (infinite-loop guard), `test_recast_rejects_startup_failures` (stream_age floor), `test_recast_requires_valid_hwm`, `test_recast_min_stream_age_is_stable` (pins the constant so it can't drift silently). **Generic lesson**: when a device-side flake is indistinguishable from a protocol-level failure, recovery beats diagnosis — one cheap retry with the right guards turns a support incident into a transparent 15 s blip. But the retry must be capped at ONE per stream or a wedged device burns cycles indefinitely, and it must respect natural EOF or it replays the credits forever.

- **Worker cleanup must not imply media cleanup** — WebTorrent/ffmpeg are owned workers with their own failure domain. Emergency cleanup should terminate stale workers only; it must not delete media or rewrite playback history. Keep this rule linked to `OPERATIONS.md`. This was added after a private production incident where abandoned WebTorrent workers exhausted host RAM/swap and made unrelated services appear wedged.
- **webtorrent `-s` FIXED** (our PR #3011, fixes #331) — piece verification bug in `_markUnverified` re-selected ALL pieces, downloading entire torrent despite `-s`. Fix: `Selections.contains()` guard prevents re-selecting deselected pieces. Patched in-place on Darwin at `~/.local/share/mise/installs/node/24.14.0/lib/node_modules/webtorrent-cli/node_modules/webtorrent/lib/`. Verified: 27-file season pack → only target file + 1.7MB boundary pieces downloaded (sparse files, no actual disk waste). Smart ranking still prefers single-file torrents as belt-and-suspenders
- **Chromecast fetches URLs itself** — never hand it `localhost`. Use the configured stream host (`stream_host`) as the source of truth for Cast-fetchable URLs; it can be a DNS name when Cast devices resolve local DNS correctly, or an IP if they do not. Do not hard-code host rewrites after the cast call, because that only changes the client response and does not protect the Chromecast.
- **Local bypass completion markers require byte evidence** — `.spela_done` should only be written when search-result size metadata is available and the physical file bytes match it. Duration is not a byte size. See `OPERATIONS.md`.
- **Transcoded streaming via axum endpoint** (Mar 20, 2026) — `python3 -m http.server` sent `Content-Length` for a growing fMP4 file → Chromecast read that many bytes, thought stream complete, stopped after ~10s. Fix: `/stream/transcode` axum endpoint with chunked transfer (no Content-Length) + `StreamType::Live` (tells Chromecast not to expect fixed length). **5MB pre-buffer** proves sustained torrent+transcode health before casting. ffmpeg PID tracked for stream tailing + cleanup
- **EAC3/AC3/DTS → AAC transcode** — ffprobe auto-detect, ffmpeg with `-re` flag (real-time pacing, never outruns download) + `-reconnect_at_eof` (handles stalls). Fragmented MP4 output (`frag_keyframe+empty_moov`) playable from first byte
- **Subtitles burned into video** — when transcoding is needed, SRT subtitles are hardcoded via `-vf subtitles=` with NVENC GPU encoding (`h264_nvenc`). Works around rust_cast's lack of Cast protocol track support. Subtitles fetched from Stremio OpenSubtitles v3 (zero auth)
- **Self-healing: dead seeds** — if 0% download progress after 12s, auto-tries next search result (up to 3 retries). Cleanup between retries (kill ffmpeg, delete partial transcoded files)
- **catt mDNS ~40% flaky** — V2 uses rust_cast native + IP cache, no Python deps
- **webtorrent --chromecast broken** — serve via HTTP (:8888) + cast URL via rust_cast
- **systemd PATH** — needs explicit PATH env for mise shims (webtorrent not in default PATH)

- **Intro clip** (Mar 21) — `~/.config/spela/intro.mp4` prepended via ffmpeg concat filter. Always plays when present (triggers NVENC pipeline). - **Pause/resume works** (Mar 26) — no stall timeout (removed). Tested up to 10-minute pauses. ffmpeg exit is the only stream termination signal
- **Silent Range requests** (Mar 26) — `/stream/transcode` honors `Range: bytes=N-` by seeking the file, but ALWAYS returns 200 (never 206/Accept-Ranges). Chromecast switches to VOD mode if it sees range headers → disconnects on growing files. Silent approach: seek works, Chromecast stays in live mode
- **Seeking does NOT work** with Default Media Receiver — fMP4 has no byte-offset index, Chromecast can't map timestamps to file positions. Both `StreamType::Live` and `StreamType::Buffered` (with known duration from ffprobe) tested — seek always resets to start. Jellyfin solves this with Custom Receiver + Shaka Player + server-side seek-restart. **Next step**: $5 Google Cast SDK → Custom Receiver App
- **Post-playback reaper** (Mar 21) — background task monitors ffmpeg PID. When movie ends (ffmpeg exits), waits 3 min grace, kills webtorrent (~1.5GB freed), cleans files. Detects stream replacement and exits cleanly

- **HLS rework gotchas — five compounding bugs** (Apr 15) — even after the architectural switch from chunked-fMP4 to HLS, the Apr 15 live debug session uncovered FIVE compounding bugs that all had to be fixed before Default Media Receiver would accept the stream. Documenting each one because a future session will hit them again on a different receiver: (1) **Segment route mismatch** (commit `2c53b4f`) — ffmpeg's HLS muxer emits relative segment URLs (`seg_00000.ts`) which Chromecast resolves against the manifest URL → `/hls/seg_00000.ts`, NOT `/hls/segment/seg_00000.ts`. Match what ffmpeg writes, not what feels semantic. (2) **fmp4 segments need `hlsSegmentFormat` in the LOAD message** — rust_cast's `Media` struct exposes only 5 fields (contentId, contentType, streamType, duration, metadata) and can't pass `hlsSegmentFormat`. CAF Receiver needs that hint to parse fmp4 segments and otherwise stays IDLE. **Use MPEG-TS segments** (`-hls_segment_type mpegts`) — Default Media Receiver auto-detects them from the manifest, no extra LOAD-message field needed. (3) **HLS v6 features break old Chromecasts** — `-hls_playlist_type event` and `-hls_flags independent_segments` both bump the manifest to version 6, which the older Shaka Player on CrKey 1.56 firmware can't parse. Confirmed live: receiver fetched `playlist.m3u8` four times in two seconds then bailed to `idle_reason=ERROR` without ever requesting a single segment. **Drop both flags** — ffmpeg's HLS muxer auto-determines the manifest version from features used, and avoiding v6 features keeps it at v3-v4. Note: `-hls_version` is NOT a valid ffmpeg HLS muxer option (it'll error out at startup with "Unrecognized option"). (4) **`StreamType::Live` is wrong for HLS** (commit `e65febf`) — for HLS playlists, the cast LOAD message must use `StreamType::Buffered`. With `Live`, Default Media Receiver acknowledges the LOAD but never fetches the manifest URL — `player_state` stays IDLE indefinitely. `cast.rs::cast_url` now infers `stream_type` from `content_type`: `application/vnd.apple.mpegurl` / `application/dash+xml` → `Buffered`, everything else → `Live`. (5) **Bare media playlist needs a master playlist wrapper with explicit CODECS** (commit `ebab5e5`) — older receivers won't load a media playlist directly via the cast LOAD URL because it lacks `CODECS` / `RESOLUTION` / `BANDWIDTH` metadata. Compared the diagnostic: spela's bare media playlist made the receiver fetch it 4× and bail; Apple's bipbop reference HLS (which has a proper master playlist with `CODECS="avc1.64001f,mp4a.40.2"`) plays in 5 seconds on the SAME device. **`/hls/master.m3u8` synthesizes a master playlist on the fly** with hardcoded CODECS for spela's standard transcode profile (H.264 High@4.0 + AAC LC stereo, 6 Mbps, 1920×1080) and references the ffmpeg-written media playlist by relative path
- **Chromecast hardcodes Google DNS, can NOT resolve LAN hostnames** (Apr 15, RESOLVED via router-side DNAT hijack) — Chromecast devices use 8.8.8.8 / 8.8.4.4 by default and completely ignore DHCP option 6 (DNS server), even when it's the LAN router handing out itself as DNS. Empirical proof caught live on Apr 15 via tcpdump on Darwin's `eno1`: `192.168.4.126.36919 > 8.8.8.8.53: A? www.google.com` (Chromecast directly querying Google on plain UDP 53, bypassing AdGuard on 192.168.4.1 entirely).   ```
  iptables -t nat -I PREROUTING -i eno1 -s 192.168.4.126 ! -d 192.168.4.1 \

           -p tcp --dport 53 -j DNAT --to-destination 192.168.4.1:53
  ```
  Scoped to 192.168.4.126 (Fredriks TV) only so no other LAN device is touched. Persisted in `/etc/iptables/rules.v4` via `netfilter-persistent`. Standard Pi-hole/AdGuard technique. Reversible in one command (`-D` instead of `-I`). spela startup still WARNs if `stream_host` looks like a hostname — the warning is a generic safety net for deployments that don't have the router-side hijack (e.g. spela running behind a non-router host). - **DMR overlay is stream-type-dependent, not metadata-dependent** (Apr 29 v3.2.1) — Default Media Receiver (DMR / app id `CC1AD845`) renders its progress-bar UI **iff the HLS stream is in VOD mode** (manifest contains `#EXT-X-ENDLIST` and/or `#EXT-X-PLAYLIST-TYPE:VOD`). Live-mode HLS (no ENDLIST) shows NO progress bar — there's no duration to scrub against. This contradicts the earlier "rich-UI metadata triggers persistent overlay" model from commits `17ef4c0`/`48aa358`: rich `Metadata::TvShow`/`Metadata::Movie` only adds the *poster + title splash*; the *seek bar overlay* is gated by VOD-vs-live detection alone. Anchoring user observation (Apr 29): "I've been able to watch The Boys episodes etc WITHOUT the seek bar in the past, never had a custom receiver" — past content played fine because spela's default HLS output is live-mode (no ENDLIST during streaming, ffmpeg appends it only after the muxer closes). The Apr 28 B1 commit `1fda4a7` (`experimental_endlist_hack`) and the Apr 29 B2 commit `98cb043` (`vod_manifest_padded`) both intentionally promoted the manifest to VOD mode to chase a different fix, and inadvertently introduced the persistent overlay regression. **Fix**: `vod_manifest_padded = false` (default in `src/config.rs:126`; was set to `true` in Darwin's local config during B2 testing, reverted Apr 29) AND `experimental_endlist_hack = false`. **Trade-offs are display-only**: (a) no total-duration display on TV (was already true in v3.0/v3.1, user described as "the way I've been watching for months"), (b) no native scrub-via-remote (already-not-supported because transcoded HLS has no byte index for arbitrary seek). HWM accuracy is NOT a downside — the "HWM corruption" worry was B1-specific (Shaka chasing a *moving* `EXT-X-ENDLIST`) which honest live mode doesn't have at all (no end marker, no drift surface). `cast_health_monitor`'s `absolute = current_time + ss_offset` math works identically in live mode and VOD mode — Smart Resume, per-episode keying, position-jump sanity checks were all developed against live-mode HLS as the v3.0/v3.1 baseline. Cast LOAD uses `StreamType::Buffered` (not `Live`) so the receiver plays sequentially from segment 0 without live-edge tracking. End-of-stream detection: ffmpeg appends `EXT-X-ENDLIST` when transcode finishes, manifest flips to VOD at that point, receiver plays remaining buffered segments and transitions to IDLE — `cast_health_monitor`'s `IDLE_FAILURE_THRESHOLD` (3 polls × 5s = 15s) catches it; reaper grace covers the rest. The proper end-state for both overlay-free *and* total-duration-display is the parked Custom Receiver path ($5 Cast SDK registration) — receiver code already exists at `static/cast-receiver.html`. **Generic lesson**: when chasing a UI symptom on a closed-source receiver, validate the model against historical user observation before committing to a fix. Two intermediate hypotheses (rich metadata, ENDLIST trick) both *technically worked* in the way they claimed to (each got DMR to render or hide the splash differently), but neither addressed the actual user-visible issue (persistent progress bar) because that was governed by a third axis (VOD vs live mode) we hadn't named. The overlay was tied to *what kind of stream* DMR thought it was watching, not *what metadata* the LOAD message advertised. Cross-ref: `~/Projects/spela/TODO.md` v3.2.0 changelog correction Apr 29
