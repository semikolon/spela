## Usage

```bash
spela search "Good Luck Have Fun Dont Die"       # Auto-detects movie vs TV
spela search "legion" --season 1 --episode 5   # Search + rank results
spela play 1                                     # Play result #1 (auto: magnet, file_index, metadata)
spela play 1 --cast "Vardagsrum"                 # Play to living room TV
spela play 1 --no-intro                          # Skip intro bumper
spela next                                       # Next episode
spela pause / resume / stop                      # Playback control
spela status / history / targets                 # Info
```

`spela play <N>` is the primary flow. Carries all metadata (file_index, IMDB, show, season, episode) automatically from the last search.

## Build & Deploy

```bash
# Repo cloned on Darwin at ~/Projects/spela (Rust 1.94):
cd ~/Projects/spela && git push   # from Mac
ssh darwin 'cd ~/Projects/spela && git pull && cargo build --release'
ssh darwin 'sudo systemctl restart spela'
```

`~/.local/bin/spela` on Darwin is a symlink to `~/Projects/spela/target/release/spela` (verified Apr 30, 2026 — `lrwxrwxrwx ... -> /home/fredrik/Projects/spela/target/release/spela`). `cargo build --release` rewrites the inode the symlink points at; the running spela process keeps the OLD inode open until `systemctl restart` re-execs against the symlink. **No `cp` step needed** — earlier versions of this doc had `cp ~/Projects/spela/target/release/spela ~/.local/bin/` which errors with "are the same file" (Apr 30 deploy hit this).

**Mac builds: use `./bin/build-mac.sh`, NOT bare `cargo build --release`** (v3.8.0, May 20, 2026). The wrapper does `cargo build --release` + ad-hoc codesign with a stable identifier (`com.fredrikbranstrom.spela`) so TCC grants (Full Disk Access for `serve-library`'s `read_dir(BOHR)`) survive rebuilds. Background: TCC keys grants to the binary's code-signature; cargo's `linker-signed` default identifier is content-derived (`spela-56c693bdcecc486f` form), so every rebuild flips the identifier and silently revokes FDA. The stable-identifier codesign is the fix per global CLAUDE.md "TCC code signature gotcha". **One-time re-grant** required after the FIRST `bin/build-mac.sh` run (when the new stable identifier replaces the old hash-based one): `System Settings → Privacy & Security → Full Disk Access → +`, add `~/Projects/spela/target/release/spela`, then `launchctl kickstart -k gui/$(id -u)/com.fredrikbranstrom.spela-library`. From the SECOND build onwards, the grant should survive — empirical verification pending across the next few rebuilds. (Darwin/Linux builds keep using bare `cargo build --release` — TCC is macOS-only.)

**Darwin working-tree drift recovery** (May 13, 2026, observed mid-deploy): if `git pull` fails on Darwin with *"cannot pull with rebase: You have unstaged changes"* AND `git diff --stat HEAD` shows ≥1000 LOC across multiple source files, do NOT `git stash`, do NOT force-discard. The likely cause is past-session edits made directly on Darwin (incident-debug hotpatches) that were later committed FROM Mac Mini, leaving Darwin's working tree dirty against an older HEAD pointer. Forensic procedure: (1) `git rev-parse HEAD origin/master` on Darwin to find Darwin's HEAD; (2) on Mac Mini, `git log --oneline <darwin-HEAD>..HEAD` to enumerate the commits Darwin doesn't have yet; (3) `sha1sum` each Darwin WT file against `git show <candidate-commit>:<file>` on Mac Mini for the same path — match means the WT content is byte-identical to a master commit, safe to discard. If all files match (or match + formatting-only deltas), `git checkout -- <files>` on Darwin is non-destructive — content is preserved in master. If even one file shows unique content, STOP and propose committing-from-Darwin to a branch instead. May 13 anchor: 8 files changed on Darwin, 7 matched `ab9a9a5` byte-for-byte, 1 (`search.rs`) was an older snapshot superseded by a `cargo fmt` pass on Mac — all features preserved in master, safe discard, deploy completed cleanly.

## Hard-Won Lessons (preemptive architecture gotchas — read before touching these areas)

Incident narratives + RCA + journal evidence: `git log` (every fix-commit carries the detailed analysis). Key SHAs: `7cd71c6` v3.4.0 bad-source trio · `c3b41a0` v3.4.1 ranker transitivity · `e0e81bf` v3.4.2 pause-gate · `e3960aa` v3.4.3 seek semantics · `40d3b96` v3.5.0 cache foundation · `ee38f92` 10-bit NVENC · `dcbaed7` CORS revert · `2c598f4` dual-bind · `245bef0` v3.8.0 serve-library drive-aware daemon. Deep dives: `docs/INCIDENT_REPORT_WILDERPEOPLE_2026_05_01.md`, `docs/CAST_DEBUGGING_RECIPES.md`.

**serve-library drive-aware daemon (v3.8.0, `library_origin.rs`)**: state machine `Initial → {Serving, Degraded, Waiting}` driven by 30 s `run_state_machine_probe` (tokio task, FS I/O in `spawn_blocking` so cold USB-HDD spin-up never stalls axum). `LibraryOriginState.healthy_roots` is `Mutex<Vec<PathBuf>>` — handlers read THIS, never `configured_roots`, so a vanished drive immediately disappears from `/library/list`. **NEVER bail on empty `healthy_roots`** — bail ONLY on empty `configured_roots` (genuine config error). `canonicalize_roots_quiet` (no per-failure WARN) replaces the old loud `canonicalize_roots` — re-running every 30 s would re-introduce the spam this refactor exists to eliminate; the transition logger handles the user-visible signal exactly once per change. `/library/{list,match}` return `503 + JSON {error, status:"waiting", missing_roots:[...]}` when waiting (honest upstream signal vs misleading 404). `/library/list` logs at DEBUG (was INFO — every SPA poll spammed). New `/health` always returns 200 + state JSON. `send_ntfy_alert` is best-effort 3 s timeout, NEVER crashes the daemon. **Subsumes v3.6.3 self-warm** — every probe touches each healthy root (30 s << macOS's ~10 min USB-HDD idle threshold), so the dedicated std-thread is gone; do NOT add it back. **The load-bearing fix is the BINARY** — once `run()` stops exit(1)ing on missing roots, launchd's `KeepAlive=true` becomes harmless. Plist tightening to `KeepAlive {Crashed:true}` + `ThrottleInterval=60` is defense-in-depth queued for a dotfiles session, NOT required to eliminate the spam class.

**The May-19 power-cut incident** (which forged v3.8.0): BOHR USB-SATA bridge wedged after the surge → not enumerating → `canonicalize_roots` filtered all roots out → `bail!` → exit 1 → launchd KeepAlive respawn @ 10 s → 23 h × 8640 cycles × ~600 B/cycle ≈ 5 MB of identical "no usable library_dirs" stderr+stdout. Recovery was a USB replug. The replug + restart was the easy half; the design-failure-as-noise-not-signal was the deeper bug — "*Make failures visible before fixable*", "*Silent accumulation*", "*Auto-resolving gates over blocking gates*" all fired simultaneously. v3.8.0 closes the class.

**Web-remote portless access (`require_host_header` / `config.allowed_hosts`)**: spela 403s any `Host` not in {loopback, `stream_host`, `config.allowed_hosts`} *before any handler* (H2 DNS-rebind defense). Front it behind a LAN reverse proxy ⇒ the proxied hostname MUST be added to `allowed_hosts` or every request is 403. A fronting health-check must target `/remote` (200) — NOT `/` (now a 307 redirect) and NOT a bare path (spela has no 200-returning `/`). `GET /` → 307 `/remote` exists so a bare host resolves portless. `allowed_hosts` lives in `~/.config/spela/config.toml`, which is a **forward-only runtime file** — spela rewrites it itself (`config.rs` `save_config`, full-file `toml::to_string_pretty`) so runtime-accrued fields (`allowed_hosts`, `remote_origins`) are NOT in any committed baseline. It is **gitignored in this repo BUT nit-tracked in dotfiles with a stale baseline** — a `git checkout`/nit-pull on Darwin **clobbers the live runtime config** (2026-05-18 prod incident: 403 + empty library). NEVER git-checkout/clobber Darwin's live spela config; recover from `~/.config/spela/config.toml.bak-<epoch>` (spela's own pre-rewrite backup) then re-add `allowed_hosts`. The systemic fix (untrack from nit, or nit forward-only-sync) is dotfiles-domain — tracked there, not here.

**HWM is stream-progress, not watch-progress**: resuming from saved HWM after a *server-side* restart (especially one you initiated) re-resumes where the **stream** was — which auto-advances during unwatched playback — NOT where the viewer actually watched. After any self-caused interruption, rewind to 0 or an explicit user-stated anchor; never silently resume HWM (spoiler risk, twice-burned). `seek_to:0.0`/`N` in the play body is the clean re-transcode-from-N path (`resumed_from:null` confirms no HWM use). See `docs/CAST_DEBUGGING_RECIPES.md` §4 HWM-pollution.

**Web-remote now-playing state machine — `/status` carries NO play-state**: do NOT reconcile `playing` from polling (the original `pollNow` reasserted `playing=true` every poll → every tap re-sent `/pause`, `/resume` permanently unreachable). `playing` is **user-intent-only** (set on play start + flipped by the pause/resume button; never touched by `/status`). And cold-start is **time-bounded**, NOT `!running`: a fast bridge/local play fully transcodes then ffmpeg EXITS (`running:false`/`process_dead`) while the Chromecast plays/pauses the COMPLETE HLS perfectly — that is an established, controllable stream, not eternal "Starting…". Gate: `!running && !everRan(sticky once running) && !ENDLIST && age<75s`.

**Web-remote phone-direct (Phase 6)**: reuses the proven **`target:"vlc"`** no-Chromecast HLS path — `do_play` gates `cast.cast_url` behind `target=="chromecast"` but still runs `transcode_hls` for non-cast, so `POST /play {…,target:"vlc"}` serves `/hls/master.m3u8` with no cast (zero backend change; do NOT add a new "browser" target — `vlc` already is it). The SPA plays it in a native `<video>` (iOS Safari HLS) / lazy CDN-pinned hls.js (Chromium). **`renderNowFull` MUST render-once-guard the `<video>`** (`if($("#phvid")) return`) — it's called every 3 s poll; without the guard the element rebuilds and playback restarts mid-stream. `pollNow` phone-guards: skip cast cold-start, `/api/position`, and the now-bar (the `<video>` IS the transport). Fresh play removes the old `#phvid` so it rebuilds.

**Ranker (`rank_results_mut`)**: 5 tiers via `effective_res_tier` — bakes seed-viability INTO resolution bucket so the comparator is a strict total order. *Pairwise threshold-fallthrough rules are a non-transitive comparator class bug.* 30× HEVC seed-disparity override (`SEED_DISPARITY_OVERRIDE`). DV demoted (NVENC GTX 1650 can't decode profile 5/7 RPU). `filter_results_by_show_title` token-based pre-ranker. `is_hevc_from_title` normalizes codec separators (`H 265`, `h.265`, `H-265` all match). Sarpetorp policy: 1080p > 720p > 480p > 2160p > unknown (TVs max 1080p native).

**cast_health_monitor (`src/server.rs`)**: identity = `started_at` not `torrent_pid` (Local Bypass pid=0 collisions). `paused_seen_in_session` HARD GATE on `should_attempt_recast` — sticky, cleared only by stream replacement; CrKey 1.56 unloads receiver app after ~16 min pause (Paused→IDLE looks like wedge). BUFFERING transient (don't kill) but bounded by `MAX_BUFFERING_DURATION_SECS=60s`. Auto-recast frequency-capped (`RECAST_COOLDOWN_SECS=30s`), not lifetime cap — receiver flakes recur every 15-30 min and each recast IS recovery. `is_idle_in_cold_start_window` protects 60s startup (DMR media-session allocation can take 25-60s). `HWM_CLEAR_FRACTION=0.96` is the ONLY threshold for HWM-clear-at-near-EOF, NOT kill — *one constant for two policies is a load-bearing coincidence*. `is_position_jump_suspicious(delta_wall, delta_abs)` rejects physically-impossible advances (stale current_time on stream replacement).

**do_play invariants (`src/server.rs`)**: NEVER reintroduce `do_cleanup` between `TorrentEngine::start` and transcode/cast step — SIGTERMs the just-spawned worker. `do_cleanup` on every error-return path after `TorrentEngine::start` (else orphans). Auto-resume from HWM via `app_state.get_position` unless explicit `--seek N` (which clears HWM — *explicit user actions override remembered state*). `should_fail_fast_stream_start(elapsed, segments)` fires at 20s/0 → `handle_play` auto-retry bumps `result_id` (distinct from cast_health_monitor cold-start IDLE — upstream-side, fires BEFORE LOAD). **My-Library tap-to-play (v3.7.1)**: a title-only `/play` (`{title,target}`, no magnet — AC-3.3) IS the library-bridge contract; the magnet is a TORRENT concern — require it ONLY at the `if !is_local` torrent fallback (`magnet_required_error`), NEVER as a do_play precondition (the precondition made the *entire* My-Library bridge structurally unreachable; "LIVE" had only ever meant list/posters — *Built≠Verified*). Frontend twin: the target-picker MUST stash an `S.pending` replay when a play is tapped before a target is set and fire it on target-select, else the first tap silently no-ops ("nothing happened, no cast icon"). **Third facet (v3.7.2)**: `do_play` auto-fills `poster_url` ONLY via the result_id/search path — a library tap has no result_id, so the SPA must forward `entry.poster_url` (it already has it from `/library`) in the `/play` body, else `CurrentStream.poster_url=None` → blank Now-view poster. Same class: every library-bridge facet is a caller shape the search-centric `do_play` didn't cover — audit new title-only callers for ALL search-only auto-fills. **Fourth facet (v3.7.3)**: the SPA scrubber's transcoded-frontier must be read from the **master's first variant URI** (`/hls/master.m3u8` → `stream_0.m3u8`), NOT a hardcoded `/hls/playlist.m3u8` — ffmpeg `-var_stream_map` multi-variant output has no `playlist.m3u8` (404 → `frontier=0`/`endlist=false` forever → `seekMax` collapses to full duration → forward seeks past the real transcoded edge silently no-op). **Local Bypass + disk safety (`src/server.rs`, `src/disk.rs`)**: `find_local_bypass_match` (title/year/quality/health decision matrix). `top_level_file_is_healthy` trusts title-match for single-file releases (≥100 MB, non-sparse). **Directory-entry health is 3-tier** (`local_bypass_file_is_healthy`, v3.6.0): (1) `expected_bytes>0` → strict ±1% size match (torrent path, season-pack disambiguation); (2) `.spela_done` marker → completed `~/media` download, non-sparse only; (3) NEITHER → curated external-library entry (serve-library / remote-origin: no torrent size, no marker) → same lenient ≥100 MB non-sparse bar as top-level. **Tier 3 is load-bearing — without it every directory-style library film is invisible to Bypass (serve-library DOA bug, v3.6.0 deploy). Do NOT collapse tier 3 back into a marker/size requirement.** `.spela_done` requires byte evidence, never duration. `dir_size` uses `metadata.blocks() * 512` via `MetadataExt` (sparse-aware — webtorrent/librqbit BEP-53 file selection creates sparse placeholders with full logical size). `MAX_MEDIA_MB=10_000` internal cap + `MIN_FS_FREE_MB=20480` host floor via `df -Pk`. `prune_to_fit` LRU-evicts oldest until under cap (refusal-wall → self-maintaining). `AppState.corrupt_files` HashSet skip-list populated by `inspect_ffmpeg_log_for_corruption` post-mortem (EBML / HEVC-ref / `dup>100` signals). **v3.6.3 remote-origin cold-HDD robustness (TWO load-bearing halves, do NOT remove either):** a remote library (serve-library) usually sits on a USB HDD that macOS spins down after ~10 min idle; a cold first `/library/match` blocks on spin-up (5-15s+). (a) `library_origin::run` spawns a self-warm std-thread touching each root immediately + every 180s so the drive never goes cold in steady state; (b) `do_play`'s remote-origin block does a cheap liveness ping (`/library/stream?h=<sentinel>` → 410 in ~1ms, no library FS touched) to tell "origin DOWN" (fast torrent fallback) from "origin UP, drive waking" (then `/library/match` with a generous 25s timeout — WAIT for the local path). Pre-v3.6.3 a bare 2s match-timeout silently fell back to torrent (wrong source + quality) on every first-after-idle play. Do NOT shrink the 25s match timeout or drop the self-warm — that resurrects the silent-torrent-fallback bug.
