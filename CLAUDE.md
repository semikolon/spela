## Overview

## Usage

```bash
spela search "Good Luck Have Fun Dont Die"       # Auto-detects movie vs TV
spela search "legion" --season 1 --episode 5   # Search + rank results
spela play 1                                     # Play result #1 (auto: magnet, file_index, metadata)
spela play 1 --cast "Vardagsrum"                 # Play to living room TV
spela play 1 --no-intro                          # Skip intro bumper
spela next                                       # Next episode
spela pause / resume / stop                      # Playback control
spela status / history / targets                 # Info
```

`spela play <N>` is the primary flow. Carries all metadata (file_index, IMDB, show, season, episode) automatically from the last search.

`~/.local/bin/spela` on Darwin is a symlink to `~/Projects/spela/target/release/spela` (verified Apr 30, 2026 — `lrwxrwxrwx ... -> /home/fredrik/Projects/spela/target/release/spela`). `cargo build --release` rewrites the inode the symlink points at; the running spela process keeps the OLD inode open until `systemctl restart` re-execs against the symlink. **No `cp` step needed** — earlier versions of this doc had `cp ~/Projects/spela/target/release/spela ~/.local/bin/` which errors with "are the same file" (Apr 30 deploy hit this).

**Build while the user is watching — compile freely, HOLD the restart (2026-07-04 operational rule)**: because the running spela keeps the OLD inode until `systemctl restart`, you can `git pull && cargo build --release` new code on Darwin *while a stream is live* with **zero interruption** — the build only stages the new binary at `target/release/spela`; nothing swaps into the running process until a restart. So during an active watch: **write + compile + commit freely; NEVER `systemctl restart spela`** (that kills the live cast) until the user confirms they're done watching. Any verification that needs a restart + a test-cast (e.g. the intro-warmup swap, or any cast-path change) waits for the user. A `cargo build` briefly competes for CPU with an in-flight transcode, but the transcode races ahead of playback and segments serve from disk, so the live stream is unaffected (`nice`/`ionice` the build if paranoid). This makes "do unattended spela coding while I watch something" a first-class workflow — the only gated step is the restart-and-verify at the end.

**Running a SECOND isolated test instance on Darwin (test streams without touching production)** — for unattended barrage/latency/health testing while production (7890/Vardagsrum) stays live. Launch:

Four collisions the env-flags avoid (each cost a failed launch when discovering this): **(1)** `SPELA_EPHEMERAL_DHT=1` — the persistent DHT reloads its stored `listen_addr` from the shared `~/.cache/com.rqbit.dht/dht.json` and tries to bind production's UDP port ("librqbit engine bootstrap failed"). This flag keeps DHT ON but skips persistence → librqbit binds an ephemeral OS-assigned port → REAL peer discovery, zero collision. (Do NOT use `SPELA_DISABLE_DHT` for latency testing — trackers-only starves peer discovery: an 859-seed torrent saw 1 peer at t=3s, inflating every measurement.) **(2)** `--port 7891` — the CLI `--port` (main.rs, default 7890) OVERRIDES `config.port`, so the `SPELA_PORT` env is dead; the loopback dual-bind also keys off it, else it fights prod's `127.0.0.1:7890`. **(3)** `SPELA_STATE_DIR`/`SPELA_MEDIA_DIR` — separate state.json + downloads (config.rs env overrides; there is NO `--state-dir`/`--media-dir` CLI arg). **(4)** `TMDB_API_KEY` — shared `~/.config/spela/config.toml` has NO key (it's in prod's systemd unit), so search 403s without it.
**⚠ Kill the test instance ONLY by its port — NEVER `pkill -f`.** Production's ExecStart is literally `spela server --host 192.168.4.1` (IDENTICAL command line; the distinguishing `--port`/env aren't in the pkill-matchable args), so `pkill -f "spela server"` KILLS PRODUCTION. Use `fuser -k 7891/tcp`. Always pass explicit `cast_name` (e.g. `"Fredriks TV"`) on every test `/play` — never rely on the shared `default_device`, which may be Vardagsrum (the user's live TV).

**Darwin working-tree drift recovery** (May 13, 2026, observed mid-deploy): if `git pull` fails on Darwin with *"cannot pull with rebase: You have unstaged changes"* AND `git diff --stat HEAD` shows ≥1000 LOC across multiple source files, do NOT `git stash`, do NOT force-discard. If even one file shows unique content, STOP and propose committing-from-Darwin to a branch instead. ## Hard-Won Lessons (preemptive architecture gotchas — read before touching these areas)

Incident narratives + RCA + journal evidence: `git log` (every fix-commit carries the detailed analysis). Key SHAs: `7cd71c6` v3.4.0 bad-source trio · `c3b41a0` v3.4.1 ranker transitivity · `e0e81bf` v3.4.2 pause-gate · `e3960aa` v3.4.3 seek semantics · `40d3b96` v3.5.0 cache foundation · `ee38f92` 10-bit NVENC · `dcbaed7` CORS revert · `2c598f4` dual-bind · `245bef0` v3.8.0 serve-library drive-aware daemon. Deep dives: `docs/INCIDENT_REPORT_WILDERPEOPLE_2026_05_01.md`, `docs/CAST_DEBUGGING_RECIPES.md`.

**serve-library drive-aware daemon (v3.8.0, `library_origin.rs`)**: state machine `Initial → {Serving, Degraded, Waiting}` driven by 30 s `run_state_machine_probe` (tokio task, FS I/O in `spawn_blocking` so cold USB-HDD spin-up never stalls axum). `LibraryOriginState.healthy_roots` is `Mutex<Vec<PathBuf>>` — handlers read THIS, never `configured_roots`, so a vanished drive immediately disappears from `/library/list`. **NEVER bail on empty `healthy_roots`** — bail ONLY on empty `configured_roots` (genuine config error). `canonicalize_roots_quiet` (no per-failure WARN) replaces the old loud `canonicalize_roots` — re-running every 30 s would re-introduce the spam this refactor exists to eliminate; the transition logger handles the user-visible signal exactly once per change. `/library/{list,match}` return `503 + JSON {error, status:"waiting", missing_roots:[...]}` when waiting (honest upstream signal vs misleading 404). `/library/list` logs at DEBUG (was INFO — every SPA poll spammed). New `/health` always returns 200 + state JSON. `send_ntfy_alert` is best-effort 3 s timeout, NEVER crashes the daemon. **Subsumes v3.6.3 self-warm** — every probe touches each healthy root (30 s << macOS's ~10 min USB-HDD idle threshold), so the dedicated std-thread is gone; do NOT add it back. **The load-bearing fix is the BINARY** — once `run()` stops exit(1)ing on missing roots, launchd's `KeepAlive=true` becomes harmless. Plist tightening to `KeepAlive {Crashed:true}` + `ThrottleInterval=60` is defense-in-depth queued for a dotfiles session, NOT required to eliminate the spam class.

**The May-19 power-cut incident** (which forged v3.8.0): BOHR USB-SATA bridge wedged after the surge → not enumerating → `canonicalize_roots` filtered all roots out → `bail!` → exit 1 → launchd KeepAlive respawn @ 10 s → 23 h × 8640 cycles × ~600 B/cycle ≈ 5 MB of identical "no usable library_dirs" stderr+stdout. Recovery was a USB replug. The replug + restart was the easy half; the design-failure-as-noise-not-signal was the deeper bug — "*Make failures visible before fixable*", "*Silent accumulation*", "*Auto-resolving gates over blocking gates*" all fired simultaneously. v3.8.0 closes the class.

**Web-remote portless access (`require_host_header` / `config.allowed_hosts`)**: spela 403s any `Host` not in {loopback, `stream_host`, `config.allowed_hosts`} *before any handler* (H2 DNS-rebind defense). Front it behind a LAN reverse proxy ⇒ the proxied hostname MUST be added to `allowed_hosts` or every request is 403. A fronting health-check must target `/remote` (200) — NOT `/` (now a 307 redirect) and NOT a bare path (spela has no 200-returning `/`). `GET /` → 307 `/remote` exists so a bare host resolves portless. `allowed_hosts` lives in `~/.config/spela/config.toml`, which is a **forward-only runtime file** — spela rewrites it itself (`config.rs` `save_config`, full-file `toml::to_string_pretty`) so runtime-accrued fields (`allowed_hosts`, `remote_origins`) are NOT in any committed baseline. It is **gitignored in this repo BUT nit-tracked in dotfiles with a stale baseline** — a `git checkout`/nit-pull on Darwin **clobbers the live runtime config** (2026-05-18 prod incident: 403 + empty library). NEVER git-checkout/clobber Darwin's live spela config; recover from `~/.config/spela/config.toml.bak-<epoch>` (spela's own pre-rewrite backup) then re-add `allowed_hosts`. The systemic fix (untrack from nit, or nit forward-only-sync) is dotfiles-domain — tracked there, not here.

**kamal-proxy `--target-timeout` MUST be ≥300s on the spela routes (2026-07-04, the months-long "won't play from the web remote" root cause)**: the web remote reaches spela via `spela.home` through **kamal-proxy**, whose `--target-timeout` (max time to wait for the target to respond to a request) defaults to **30s**. spela's local-file Chromecast `/play` legitimately BLOCKS for ~50-70s — it runs the transcode + waits for the race-ahead cast gate (a real ~120s primary-variant pre-buffer) BEFORE the HTTP request returns. So the proxy 504'd the connection at 30s → axum cancelled `do_play`'s future mid-cast → ffmpeg orphaned, no cast, the SPA fell to "Nothing playing." **This evaded diagnosis for MONTHS** because every `curl darwin.home:7890/...` or `localhost:7890/...` diagnostic goes DIRECT to spela and bypasses the proxy → always worked → masked the proxy as the culprit (the classic "works in test, fails in prod" where prod = through-the-proxy). Fix: `kamal-proxy deploy spela --host spela.home --target 192.168.4.1:7890 --health-check-path /remote --target-timeout 300s` (+ same on `spela-https`). Tracked in the renewal hook (`system/roles/router/.../kamal-proxy-spela-https.sh`) + `home_network_cheatsheet_2026_02.md` + dotfiles `TODO.md` so a container recreate / cert renewal can't silently revert it to 30s. *Generic lesson: any app whose HTTP handler legitimately blocks longer than the fronting proxy's response/target timeout will be silently aborted mid-operation — and DIRECT-to-app tests will never reproduce it. When "works when I curl it but not from the browser," suspect the proxy timeout FIRST.* The proper long-term fix (make `/play` return fast + run the cast in a detached task the HTTP lifecycle can't cancel — the warmup UI already polls `/progress`+`/status` for this) is noted in TODO as the architectural improvement; the timeout bump is the robust interim.

**HWM is stream-progress, not watch-progress**: resuming from saved HWM after a *server-side* restart (especially one you initiated) re-resumes where the **stream** was — which auto-advances during unwatched playback — NOT where the viewer actually watched. After any self-caused interruption, rewind to 0 or an explicit user-stated anchor; never silently resume HWM (spoiler risk, twice-burned). `seek_to:0.0`/`N` in the play body is the clean re-transcode-from-N path (`resumed_from:null` confirms no HWM use). See `docs/CAST_DEBUGGING_RECIPES.md` §4 HWM-pollution.

**Scrubber shows LIVE position, NOT the resume HWM (`/api/position` → `live_position` cell, not `get_position`)** (2026-07-07, Spider-Noir S01E02 backward-seek snap-back): `save_position_smart` is strict max-only HWM (`if t > current`), so it can never reflect a backward seek — a scrubber reading `get_position()` snaps the thumb back to the furthest-watched point on every poll (dragging back from 26min kept jumping to 26min). Fix: `ServerState.live_position` (in-memory `Mutex`) holds the true current absolute position, updated by `cast_health_monitor` every poll AND immediately by `handle_seek`; `/api/position` (consumed ONLY by the scrubber) returns it, title-matched, falling back to the HWM. **Do NOT route the scrubber back through `get_position` / the HWM** — that IS the bug. Resume/auto-resume stay on `get_position` (HWM) by design. Cleared in `do_cleanup`; title-guard blocks cross-stream leak.
**Web-remote now-playing state machine — `/status` carries NO play-state**: do NOT reconcile `playing` from polling (the original `pollNow` reasserted `playing=true` every poll → every tap re-sent `/pause`, `/resume` permanently unreachable). `playing` is **user-intent-only** (set on play start + flipped by the pause/resume button; never touched by `/status`). And cold-start is **time-bounded**, NOT `!running`: a fast bridge/local play fully transcodes then ffmpeg EXITS (`running:false`/`process_dead`) while the Chromecast plays/pauses the COMPLETE HLS perfectly — that is an established, controllable stream, not eternal "Starting…". Gate: `!running && !everRan(sticky once running) && !ENDLIST && age<75s`.

**Web-remote phone-direct (Phase 6)**: reuses the proven **`target:"vlc"`** no-Chromecast HLS path — `do_play` gates `cast.cast_url` behind `target=="chromecast"` but still runs `transcode_hls` for non-cast, so `POST /play {…,target:"vlc"}` serves `/hls/master.m3u8` with no cast (zero backend change; do NOT add a new "browser" target — `vlc` already is it). The SPA plays it in a native `<video>` (iOS Safari HLS) / lazy CDN-pinned hls.js (Chromium). **`renderNowFull` MUST render-once-guard the `<video>`** (`if($("#phvid")) return`) — it's called every 3 s poll; without the guard the element rebuilds and playback restarts mid-stream. `pollNow` phone-guards: skip cast cold-start, `/api/position`, and the now-bar (the `<video>` IS the transport). Fresh play removes the old `#phvid` so it rebuilds. A failed play routes to `showPlayError` → a **persistent** error card on the now-view with "Tap to retry" (re-fires the same play) + "Back to search" — NOT a toast-that-bounces (UX doc §6/§9 toast-only-critical anti-pattern); cleared on a fresh attempt / stop / retry.

**Browser HEVC pass-through (`transcode_hls_passthrough`, v3.16→v3.17, VERIFIED end-to-end 2026-07-13)**: ANY HEVC source to a browser (`target != "chromecast"`) is stream-COPIED into fMP4/CMAF — full res + bitrate, no NVENC, no downscale — instead of re-encoded; the browser decodes HEVC via MSE. **Gate (v3.17) = `target != chromecast && video hevc && no intro && seek<=0`** — the invariant is **HEVC→browser, NOT "4K"**, and it does NOT depend on `source_height` (a still-downloading torrent stream yields codec but often not height; and `transcode_hls` copies HEVC into MPEG-TS which a browser CANNOT play → 0 segments → 20s fail-fast, so EVERY HEVC-to-browser play must take the fmp4 path regardless of resolution). **Subtitles are DROPPED here** (burn-in needs a filter/re-encode; the old `sub_path.is_none()` gate shunted 4K-with-subs into an NVENC transcode that choked → fail-fast). Intro concat + mid-stream seek are the only things that still force the re-encode path. Load-bearing gotchas (all handled — do NOT regress): fMP4 segments (`.m4s` + `init.mp4`) are MANDATORY, mpegts can't carry HEVC to a browser; `-tag:v hvc1` (not hev1) or MSE rejects it; `-master_pl_name` makes ffmpeg WRITE master.m3u8 with CODECS derived from the copied stream, and `handle_hls_master` serves the ffmpeg-written master **after normalizing the codec string** (see next entry); `-hls_playlist_type event` starts the player at 0; `handle_hls_playlist` SKIPS VOD-padding when `init.mp4` exists; prebuffer + `count_hls_segments` follow `HlsTranscodeInfo.fmp4` → `.m4s`/`.ts`. Verified: Chrome (hls.js 1.6, MSE) decoded a real 2160p HEVC at 3840×1920, currentTime advancing.

**HEVC master CODECS — strip the `.B01` constraint-flags tail (`normalize_hls_master_codecs`, v3.17)**: ffmpeg's `-master_pl_name` writes the HEVC CODECS as `hvc1.2.4.L150.B01`, but Chrome's MSE `isTypeSupported()` returns **false** for that constraint-flags tail (while `hvc1.2.4.L150.90` and `hvc1.2.4.L150` are both **true**) → hls.js rejects the whole manifest (`manifestIncompatibleCodecsError`) and HEVC won't play in-browser. `handle_hls_master` reads the generated master and strips a trailing `.B..` component from any hvc1/hev1 codec token before serving (the `init.mp4` hvc1 box is untouched, so the decoder still gets the true profile). Regression tests: `normalize_strips_hevc_constraint_flags_tail`, `normalize_leaves_h264_and_hevc_without_flags_untouched`. Do NOT serve the ffmpeg master verbatim for HEVC.

**ffprobe field order — classify streams at `[/STREAM]`, never at `codec_name=` (`detect_codecs`, v3.17)**: ffprobe emits `codec_name=` BEFORE `codec_type=` within a `[STREAM]`, so the old parser (which recorded the codec only if `current_type` was already "video"/"audio") left `video_codec` + `audio_codec` **silently None/empty for every file**. Latent for months until the HEVC-pass-through gate started routing on `video_codec==hevc` — the check failed, so every HEVC play fell into the mpegts/NVENC path. Fix: buffer per-stream fields (`cur_codec`/`cur_type`/`cur_height`/`cur_lang`) and finalize at `[/STREAM]` — fully order-independent. Symptom that fingerprints this class: `Preferred audio: … codec=` logs an EMPTY codec.

**Torrent "seeds" is a claim, not deliverable capacity — do NOT read a low byte-rate as a spela bug**: Torrentio's seed count is aggregated tracker-scrape data (peers that *announced* "I have it", summed across trackers, cached, stale up to the announce interval). It massively overstates peers that will actually serve you: ghosts (client closed, not yet expired), NAT-unconnectable seeders (no inbound), and choked/slot-full peers all inflate it. The only trustworthy signal is BYTE DELIVERY (spela's probe waits for real bytes, not the count). 2026-07-13 anchor: HotD S03E04's 4K — a "344-seed" non-DV release connected 3 real peers at 0 B/s; the DV ones connected 1 peer at 0 B/s. Healthy 1080p swarms deliver fine, so this is swarm reality, not spela. When "why won't this 4K download" surfaces, check byte delivery + move on; the structural cure is debrid (TorBox, parked), which holds the file so "seeds" stops mattering.

**Ranker (`rank_results_mut`)**: 5 tiers via `effective_res_tier` — bakes seed-viability INTO resolution bucket so the comparator is a strict total order. *Pairwise threshold-fallthrough rules are a non-transitive comparator class bug.* 30× HEVC seed-disparity override (`SEED_DISPARITY_OVERRIDE`). DV demoted (NVENC GTX 1650 can't decode profile 5/7 RPU). `filter_results_by_show_title` token-based pre-ranker. `is_hevc_from_title` normalizes codec separators (`H 265`, `h.265`, `H-265` all match). Sarpetorp policy: 1080p > 720p > 480p > 2160p > unknown (TVs max 1080p native).

**TMDB candidate scoring (`tmdb_search`, `tmdb_auto_detect`, 2026-06-09 `fbf56c1`)**: short queries previously picked `results.first()` — TMDB orders by popularity regardless of title match (canonical bug: `Spring` → `Send Help (2024)` instead of `Spring (2014)`). Now `score_tmdb_candidate` = `title_token_score + year_bonus(±1) + vote_count_bonus`, with `pick_best_tmdb_candidate` applying a 0.50 floor (exact year-match overrides). `extract_year_from_query` strips a 4-digit year (bare or parenthesised) and the typed endpoints pass it as `&year=` (movie) / `&first_air_date_year=` (tv) for precision. Fallback to first-result preserved for genuine zero-overlap cases. *Generic lesson: when a list endpoint orders by popularity, scoring + a confidence floor beats first-result for any short / ambiguous query.*

**cast_health_monitor (`src/server.rs`)**: identity = `started_at` not `torrent_pid` (Local Bypass pid=0 collisions). `paused_seen_in_session` HARD GATE on `should_attempt_recast` — sticky, cleared only by stream replacement; CrKey 1.56 unloads receiver app after ~16 min pause (Paused→IDLE looks like wedge). BUFFERING transient (don't kill) but bounded by `MAX_BUFFERING_DURATION_SECS=60s`.
