# spela — AI-Agent-Ready Media Controller

## NO PERSONAL DATA (MANDATORY)
**Public repo.** Never commit real names, IPs, API keys, device names, or household details to code/docs. Use placeholders. Runtime config (`config.toml`, `state.json`) is user-local and gitignored.

## Overview

Rust CLI + HTTP API server for torrent-to-Chromecast streaming. Single 5.2MB binary, zero runtime deps on target (except webtorrent-cli and ffmpeg).

**Status**: v3.0.0 deployed on Darwin (Apr 15, 2026). HLS cast pipeline, Smart Resume, self-healing disk cache, 4-tier ranker with DV demotion, cast_health_monitor, DNS DNAT hijack for Chromecast LAN hostname resolution. Seeking via server-side restart works (native seeking via Custom Receiver App still blocked on $5 registration).

**Architecture**: `spela server` on Darwin (`darwin.home:7890`). `spela <command>` thin HTTP client from any LAN machine. Voice assistants call HTTP API directly.

## Usage

```bash
spela search "Good Luck Have Fun Dont Die"       # Auto-detects movie vs TV
spela search "legion" --season 1 --episode 5   # Search + rank results
spela play 1                                     # Play result #1 (auto: magnet, file_index, metadata)
spela play 1 --cast "Vardagsrum"                 # Play to living room TV
spela play 1 --no-intro                          # Skip intro bumper
spela next                                       # Next episode
spela pause / resume / stop                      # Playback control
spela status / history / targets                 # Info
```

`spela play <N>` is the primary flow. Carries all metadata (file_index, IMDB, show, season, episode) automatically from the last search.

## Key Files

- `src/main.rs` — CLI (clap) + server startup. Detects `play 1` (result ID) vs `play magnet:...`
- `src/server.rs` — axum HTTP API, 21 endpoints (including Custom Receiver: cast-config, seek-restart, position, retry). Orchestrates search→play→cast pipeline
- `src/search.rs` — TMDB + Torrentio. **Auto-detect movie vs TV** via TMDB multi-search. **4-tier ranking**: single-file > H.264 (≥5 seed threshold) > non-Dolby-Vision > more seeds. DV demotion is intra-codec-class — handles GTX 1650 NVENC's DV profile 5/7 RPU parse failure. **Open gap**: no resolution preference tier — a well-seeded 720p will outrank a moderately-seeded 1080p because the last tier is raw seed count. See `TODO.md` § "Open Loops" for the planned fix (tier 4: 1080p > 720p > 480p above the seed tiebreak)
- `src/cast.rs` — Native Chromecast via rust_cast + mdns-sd. 3x retry + IP cache + known device fallback
- `src/torrent.rs` — webtorrent-cli subprocess management with `-s <fileIdx>` for file selection
- `src/transcode.rs` — ffprobe codec+duration detection, ffmpeg audio transcode (EAC3/DTS/AC3→AAC) + video transcode (HEVC→H.264 via NVENC) + intro concat
- `src/subtitles.rs` — Stremio OpenSubtitles v3 (zero auth), SRT→WebVTT
- `src/disk.rs` — two-layer disk safety: 10 GB internal media cap (sparse-aware via `metadata.blocks() * 512`) + 20 GB host-filesystem free-space floor (via `df -Pk`) + age-based prune (top-level files AND directories) + **LRU pressure eviction via `prune_to_fit`** so the cap is a self-maintaining upper bound instead of a refusal wall (Apr 15, 2026). `title_matches_active` tokenizes filename vs active title so `The.Boys.S05E03.FLUX.mkv` correctly matches `The Boys S05E03`. See `OPERATIONS.md` for the full defense-in-depth map
- `src/state.rs` — state.json + last_search.json (play-by-id) + resume_positions (per-episode keyed for TV shows via `resume_position_key` + `extract_se_suffix` — Apr 15, 2026 fix for IMDb-ID collision across episodes of the same show)
- `static/cast-receiver.html` — Custom Cast Receiver (Shaka Player + CAF v3, ~200 lines)
- `src/config.rs` — `~/.config/spela/config.toml`. **Hardcoded XDG path on every platform** because Rust's `dirs::config_dir()` returns `~/Library/Application Support` on macOS, which silently hid the Mac CLI's user config and dropped it back to `localhost:7890`
- `OPERATIONS.md` — worker safety plan, stale WebTorrent prevention, emergency cleanup rules

## Hard-Won Lessons

- **Worker cleanup must not imply media cleanup** — WebTorrent/ffmpeg are owned workers with their own failure domain. Emergency cleanup should terminate stale workers only; it must not delete media or rewrite playback history. Keep this rule linked to `OPERATIONS.md`. This was added after a private production incident where abandoned WebTorrent workers exhausted host RAM/swap and made unrelated services appear wedged.
- **webtorrent `-s` FIXED** (our PR #3011, fixes #331) — piece verification bug in `_markUnverified` re-selected ALL pieces, downloading entire torrent despite `-s`. Fix: `Selections.contains()` guard prevents re-selecting deselected pieces. Patched in-place on Darwin at `~/.local/share/mise/installs/node/24.14.0/lib/node_modules/webtorrent-cli/node_modules/webtorrent/lib/`. Verified: 27-file season pack → only target file + 1.7MB boundary pieces downloaded (sparse files, no actual disk waste). Smart ranking still prefers single-file torrents as belt-and-suspenders
- **Chromecast fetches URLs itself** — never hand it `localhost`. Use the configured stream host (`stream_host`) as the source of truth for Cast-fetchable URLs; it can be a DNS name when Cast devices resolve local DNS correctly, or an IP if they do not. Do not hard-code host rewrites after the cast call, because that only changes the client response and does not protect the Chromecast.
- **Local bypass completion markers require byte evidence** — `.spela_done` should only be written when search-result size metadata is available and the physical file bytes match it. Duration is not a byte size. See `OPERATIONS.md`.
- **Transcoded streaming via axum endpoint** (Mar 20, 2026) — `python3 -m http.server` sent `Content-Length` for a growing fMP4 file → Chromecast read that many bytes, thought stream complete, stopped after ~10s. Fix: `/stream/transcode` axum endpoint with chunked transfer (no Content-Length) + `StreamType::Live` (tells Chromecast not to expect fixed length). **5MB pre-buffer** proves sustained torrent+transcode health before casting. ffmpeg PID tracked for stream tailing + cleanup
- **EAC3/AC3/DTS → AAC transcode** — ffprobe auto-detect, ffmpeg with `-re` flag (real-time pacing, never outruns download) + `-reconnect_at_eof` (handles stalls). Fragmented MP4 output (`frag_keyframe+empty_moov`) playable from first byte
- **Subtitles burned into video** — when transcoding is needed, SRT subtitles are hardcoded via `-vf subtitles=` with NVENC GPU encoding (`h264_nvenc`). Works around rust_cast's lack of Cast protocol track support. Subtitles fetched from Stremio OpenSubtitles v3 (zero auth)
- **Self-healing: dead seeds** — if 0% download progress after 12s, auto-tries next search result (up to 3 retries). Cleanup between retries (kill ffmpeg, delete partial transcoded files)
- **catt mDNS ~40% flaky** — V2 uses rust_cast native + IP cache, no Python deps
- **webtorrent --chromecast broken** — serve via HTTP (:8888) + cast URL via rust_cast
- **systemd PATH** — needs explicit PATH env for mise shims (webtorrent not in default PATH)
- **HEVC/VP9/AV1 auto-transcode** (Mar 26) — `detect_codecs()` returns video+audio+duration. HEVC torrents auto-transcoded to H.264 via NVENC. Search ranking deprioritizes HEVC (but well-seeded HEVC beats dead H.264, threshold ≥5 seeds)
- **Intro clip** (Mar 21) — `~/.config/spela/intro.mp4` prepended via ffmpeg concat filter. Always plays when present (triggers NVENC pipeline). Both streams scaled to 1080p. 45s pre-buffer (vs 25s without intro). `--no-intro` to disable. **Gotcha**: `-dn -map_metadata -1` required — concat produces `bin_data` streams Chromecast rejects
- **Pause/resume works** (Mar 26) — no stall timeout (removed). Tested up to 10-minute pauses. ffmpeg exit is the only stream termination signal
- **Silent Range requests** (Mar 26) — `/stream/transcode` honors `Range: bytes=N-` by seeking the file, but ALWAYS returns 200 (never 206/Accept-Ranges). Chromecast switches to VOD mode if it sees range headers → disconnects on growing files. Silent approach: seek works, Chromecast stays in live mode
- **Seeking does NOT work** with Default Media Receiver — fMP4 has no byte-offset index, Chromecast can't map timestamps to file positions. Both `StreamType::Live` and `StreamType::Buffered` (with known duration from ffprobe) tested — seek always resets to start. Jellyfin solves this with Custom Receiver + Shaka Player + server-side seek-restart. **Next step**: $5 Google Cast SDK → Custom Receiver App
- **Post-playback reaper** (Mar 21) — background task monitors ffmpeg PID. When movie ends (ffmpeg exits), waits 3 min grace, kills webtorrent (~1.5GB freed), cleans files. Detects stream replacement and exits cleanly
- **GPU coexistence** — NVENC transcode (163MB), llama.cpp embeddings (2.8GB), Chrome kiosk (103MB) all fit in 4GB VRAM simultaneously
- **Jellyfin evaluated, rejected** (Mar 26) — library-centric (no "stream torrent NOW" flow), C#/.NET plugins, mixed Chromecast reliability. spela Custom Receiver is cleaner path for seeking
- **Custom Cast Receiver built** (Mar 26) — `static/cast-receiver.html` (Shaka Player + CAF v3). Self-configures via `/api/cast-config` because **rust_cast's Media struct only has 5 fields** (contentId, streamType, contentType, metadata, duration) — no tracks, customData, or textTrackStyle. Server endpoints: `/cast-receiver.html`, `/cast-receiver/intro.mp4`, `/cast-receiver/subs.vtt`, `/api/cast-config`, `/api/seek-restart`, `/api/position`, `/api/retry`. Blocked on $5 Cast SDK registration. Spec at `.claude/specs/custom-cast-receiver/`
- **Cast SDK terms** (Mar 26) — reviewed, no restrictions on content type or personal/commercial use. ToS governs SDK usage (user-initiated casting, no persistent receiver storage), not content. Safe for distribution
- **ffmpeg zombie fix** (Mar 26) — `std::mem::forget(child)` prevented `waitpid()`, creating zombie processes. Fix: `tokio::spawn(child.wait())` reaps immediately on exit

Other Stremio addons (MediaFusion, Knightcrawler, Comet) require encrypted config URLs or debrid service — not usable without self-hosting. Future: self-hosted Jackett on Darwin for private trackers

- **`do_cleanup` mid-flight kill** (Apr 15) — `do_play` had a misplaced `do_cleanup(&state)` between `start_webtorrent` and the transcode/cast step. `do_cleanup → stop_by_pid_file → kill_all_webtorrent()` SIGTERM'd its own just-started worker every time, so ffmpeg always hit `Connection refused` at `darwin.home:8888`. Removed in `4d3ef73`. Pre-start cleanup at the top of `do_play` (lines 387-397) was already sufficient. **NEVER reintroduce `do_cleanup` between worker spawn and reaper spawn.** Audit notes in `OPERATIONS.md` defense plan
- **Cast-failure cleanup defense** (Apr 15) — pre-existing leak: `do_play` returned errors from `cast_url()` failures without cleaning up the freshly-started workers, leaving them as orphans until next play / `kill-workers` / restart. Layer-8 defense added in `8735ea4`: `do_cleanup(&state)` on both cast-failure return branches. The reaper hasn't been spawned yet at that point, so explicit cleanup is the only thing standing between a cast hiccup and the Apr 8 incident class
- **`dir_size` is sparse-aware** (Apr 15) — webtorrent `-s <idx>` creates sparse placeholder files for unselected files in a torrent: `metadata.len()` reports the full multi-GB torrent size while only a few KB actually exist on disk. The buggy `dir_size` summed `len()` and tripped the 10 GB cap before any real download. Fix in `9f58307`: `metadata.blocks() * 512` via `std::os::unix::fs::MetadataExt`. Regression test `test_dir_size_counts_sparse_file_as_allocated_not_logical` pins it. Local Bypass already had its own sparse check (physical < logical → reject), only `dir_size` was missing it
- **Host filesystem safety floor** (Apr 15) — independent secondary cap protecting Darwin's router/runtime/database services from a runaway torrent. spela refuses new plays when `df -Pk ~/media/` reports <20 GB free, even when its own 10 GB media cap is satisfied. Best-effort (None on `df` failure) so a parser regression can't make spela unusable. `disk.rs::check_space`, commit `688bab5`. Complements rather than replaces the internal media cap
- **Mac CLI XDG path hardcoded** (Apr 15) — `dirs::config_dir()` returns `~/Library/Application Support` on macOS, hiding the Mac Mini's real `~/.config/spela/config.toml` and silently dropping the CLI back to `server = "localhost:7890"`. Hardcoded `dirs::home_dir().join(".config").join("spela").join("config.toml")` in `config.rs::config_path`, commit `267792f`. Regression test `test_config_path_uses_xdg_on_all_platforms` pins it. Same gotcha applies to any cross-platform Rust app using the `dirs` crate
- **Dolby Vision HEVC is NVENC-broken on GTX 1650** (Apr 15) — DV profile 5/7 RPU NAL units fail to parse. ffmpeg spams `Error parsing DOVI NAL unit` / `RPU validation failed: 0 <= el_bit_depth_minus8 = 32 <= 8` on every frame and the live transcode collapses to ~0.937x realtime (unviable for streaming). Search ranker now demotes DV titles below their non-DV siblings (4th tier). `has_dolby_vision_in_title()` is token-based (split on non-alphanumerics) to avoid matching DVD/DVDRip/DVD9. Commit `8313625`. Until/unless we add a `--strip-dolby-vision` ffmpeg pre-pass, DV titles are functionally incompatible
- **Ruby's `run_spela` defaults to `play 1`** (Apr 15) — Gemini was inventing the result number from her own preference for shiny labels, picking 2160p HDR DV HEVC as result #2 because "4K HDR" sounded better. Both layers of the prompt (`SPELA_TOOL_DECLARATION` in `~/.claude/hooks/conversation_engine.py` and the system tool-usage block) now say "default to args='1' — spela's ranker already returned its top pick, only override when the user explicitly asked for a specific quality/resolution/source". Dotfiles commit `916b179`. Spela's authority on which torrent to play, Ruby's job is to pass it through
- **Local Bypass top-level files + title-trust** (Apr 15) — two cascading bugs in Local Bypass that left a fully-downloaded 4.2 GB FLUX file invisible to bypass: (1) the loop only descended into directories, never inspecting top-level single-file releases (commit `664b55e`); (2) even after the loop saw the FLUX file, `is_physically_full` rejected it for being 311 MB smaller than the search result's expected size (4.19 GB FLUX vs 4.50 GB DUAL.5.1 — same content, different remux). Added `top_level_file_is_healthy` (commit `3864fd1`) that trusts the title/year/quality match for content identity and only sanity-checks the file is ≥100 MB and non-sparse. Directory bypass keeps the strict size match because it has to disambiguate between multiple files in a season pack; top-level bypass doesn't have that ambiguity
- **`cast_health_monitor` background task** (Apr 15, commit `dd111ee`) — after `cast_url()` returns OK, rust_cast drops its connection to the Chromecast and spela has zero visibility into the device's actual playback state. Status reports its local intent ("running: true, status: streaming") even when the receiver loaded the LOAD message but `player_state` never reached Playing/Buffering ("blue cast icon" failure mode). New background tokio task spawned alongside the post-playback reaper: 10s grace, polls `cast.get_info()` every 5s, after 3 consecutive Idle/Unknown/error polls declares the cast DEAD, runs `do_cleanup`, and exits. Identity uses `started_at` (DateTime<Utc>) rather than the reaper's `webtorrent_pid` because Local Bypass plays use pid=0 and back-to-back local plays would otherwise have indistinguishable identities. Catches the third class of cast failure that neither the do_play cast-failure cleanup nor the post-playback reaper handle: cast_url succeeded but the TV is stuck on the blue cast icon
- **Chromecast Default Media Receiver rejects spela's chunked-transfer fMP4** (Apr 15, RESOLVED via HLS rework) — `cast_health_monitor` confirmed live: `player_state=IDLE` for The.Boys.S05E01 even with healthy ffmpeg + perfect Local Bypass + cast_url returning OK. Root cause: spela's `/stream/transcode` endpoint serves a growing fragmented MP4 with `Transfer-Encoding: chunked`, no `Content-Length`, and always returns `200 OK` (never `206 Partial Content`). Default Media Receiver's MP4 parser doesn't accept that combo and drops to IDLE without any error surface. Confirmed against [Igalia/cog#463](https://github.com/Igalia/cog/issues/463): "media framework requires either a known content length or complete data delivery, not incremental chunked delivery". **Resolved Apr 15, 2026 by the HLS rework** — `/hls/master.m3u8` + `/hls/playlist.m3u8` + `/hls/{segment}.ts` endpoints with proper Content-Length + 206 Range. Live test against Fredriks TV showed `state=PLAYING pos=9.333` within 2 seconds of LOAD. Five compounding sub-bugs had to be fixed before HLS could play; the full chain is in the "HLS rework gotchas" entry below
- **HLS rework gotchas — five compounding bugs** (Apr 15) — even after the architectural switch from chunked-fMP4 to HLS, the Apr 15 live debug session uncovered FIVE compounding bugs that all had to be fixed before Default Media Receiver would accept the stream. Documenting each one because a future session will hit them again on a different receiver: (1) **Segment route mismatch** (commit `2c53b4f`) — ffmpeg's HLS muxer emits relative segment URLs (`seg_00000.ts`) which Chromecast resolves against the manifest URL → `/hls/seg_00000.ts`, NOT `/hls/segment/seg_00000.ts`. Match what ffmpeg writes, not what feels semantic. (2) **fmp4 segments need `hlsSegmentFormat` in the LOAD message** — rust_cast's `Media` struct exposes only 5 fields (contentId, contentType, streamType, duration, metadata) and can't pass `hlsSegmentFormat`. CAF Receiver needs that hint to parse fmp4 segments and otherwise stays IDLE. **Use MPEG-TS segments** (`-hls_segment_type mpegts`) — Default Media Receiver auto-detects them from the manifest, no extra LOAD-message field needed. (3) **HLS v6 features break old Chromecasts** — `-hls_playlist_type event` and `-hls_flags independent_segments` both bump the manifest to version 6, which the older Shaka Player on CrKey 1.56 firmware can't parse. Confirmed live: receiver fetched `playlist.m3u8` four times in two seconds then bailed to `idle_reason=ERROR` without ever requesting a single segment. **Drop both flags** — ffmpeg's HLS muxer auto-determines the manifest version from features used, and avoiding v6 features keeps it at v3-v4. Note: `-hls_version` is NOT a valid ffmpeg HLS muxer option (it'll error out at startup with "Unrecognized option"). (4) **`StreamType::Live` is wrong for HLS** (commit `e65febf`) — for HLS playlists, the cast LOAD message must use `StreamType::Buffered`. With `Live`, Default Media Receiver acknowledges the LOAD but never fetches the manifest URL — `player_state` stays IDLE indefinitely. `cast.rs::cast_url` now infers `stream_type` from `content_type`: `application/vnd.apple.mpegurl` / `application/dash+xml` → `Buffered`, everything else → `Live`. (5) **Bare media playlist needs a master playlist wrapper with explicit CODECS** (commit `ebab5e5`) — older receivers won't load a media playlist directly via the cast LOAD URL because it lacks `CODECS` / `RESOLUTION` / `BANDWIDTH` metadata. Compared the diagnostic: spela's bare media playlist made the receiver fetch it 4× and bail; Apple's bipbop reference HLS (which has a proper master playlist with `CODECS="avc1.64001f,mp4a.40.2"`) plays in 5 seconds on the SAME device. **`/hls/master.m3u8` synthesizes a master playlist on the fly** with hardcoded CODECS for spela's standard transcode profile (H.264 High@4.0 + AAC LC stereo, 6 Mbps, 1920×1080) and references the ffmpeg-written media playlist by relative path
- **Chromecast hardcodes Google DNS, can NOT resolve LAN hostnames** (Apr 15, RESOLVED via router-side DNAT hijack) — Chromecast devices use 8.8.8.8 / 8.8.4.4 by default and completely ignore DHCP option 6 (DNS server), even when it's the LAN router handing out itself as DNS. Empirical proof caught live on Apr 15 via tcpdump on Darwin's `eno1`: `192.168.4.126.36919 > 8.8.8.8.53: A? www.google.com` (Chromecast directly querying Google on plain UDP 53, bypassing AdGuard on 192.168.4.1 entirely). AdGuard's `/opt/AdGuardHome/data/querylog.json` showed zero queries from 192.168.4.126 but plenty from other LAN clients. Confirmed by multiple Google Nest Community threads going back to 2017. **Fix**: transparent iptables PREROUTING DNAT on Darwin (the active router) intercepts outbound port 53 from the Chromecast's IP and redirects it to AdGuard locally:
  ```
  iptables -t nat -I PREROUTING -i eno1 -s 192.168.4.126 ! -d 192.168.4.1 \
           -p udp --dport 53 -j DNAT --to-destination 192.168.4.1:53
  iptables -t nat -I PREROUTING -i eno1 -s 192.168.4.126 ! -d 192.168.4.1 \
           -p tcp --dport 53 -j DNAT --to-destination 192.168.4.1:53
  ```
  Scoped to 192.168.4.126 (Fredriks TV) only so no other LAN device is touched. Persisted in `/etc/iptables/rules.v4` via `netfilter-persistent`. After the hijack, `stream_host = "darwin.home"` works transparently — the Chromecast thinks it's talking to 8.8.8.8, conntrack rewrites to AdGuard, AdGuard honors the `darwin.home → 192.168.4.1` rewrite, reply source is rewritten back to 8.8.8.8 and the receiver accepts it. Standard Pi-hole/AdGuard technique. Reversible in one command (`-D` instead of `-I`). spela startup still WARNs if `stream_host` looks like a hostname — the warning is a generic safety net for deployments that don't have the router-side hijack (e.g. spela running behind a non-router host). If adding more Chromecasts to the LAN, duplicate the rule with their IPs. Backup of pre-hijack `/etc/iptables/rules.v4` saved as `.bak-2026-04-15`. Full Darwin iptables state should be hemma-managed — currently not (TODO)
- **Engine-side confirm-before-streaming** (Apr 15, dotfiles `9804bf8`) — Ruby's old "soft rule" prompt directive ("your text response MUST include a confirmation BEFORE the play tool call") was repeatedly misinterpreted by Gemini as "stop after the confirmation, do not call the tool", leaving the cast pipeline never started. Replaced with engine-level enforcement in `conversation_engine.py`: `_execute_spela` caches the JSON of every successful `spela search`, then on `spela play <N>` it parses the cached result, builds a natural-language confirmation via `_build_play_confirmation` (token-based show-name extraction + SxxExx parsing + small-number spelling), and synthesizes/plays a TTS in a daemon thread BEFORE running the spela subprocess. Best-effort: any failure silently falls through to the spela call. Lazy imports of `tts_synthesis` and `tts_daemon` avoid the circular dependency at module load time. The user hears "Got it — The Boys, season five, episode one, on your Chromecast" within ~2s of the play tool call, well before the cast pipeline completes
- **Smart Resume via `ss_offset`** (Apr 15) — `CurrentStream.ss_offset: f64` (`#[serde(default)]`) records the `-ss N` value passed to ffmpeg at transcode start, because Chromecast's `current_time` is relative to the transcoded stream (always starts at 0) not the source. Absolute source-timeline position = `chromecast_time + ss_offset`. `cast_health_monitor` snapshots ss_offset at monitor start and saves absolute position every 30s via `AppState::save_position_smart` (high-water-mark + 92% completion reset clears the entry). On next play, `do_play` auto-resumes from the stored position unless the user passes explicit `--seek`. Tests: `test_current_stream_ss_offset_roundtrip`, `test_absolute_position_translation`, `test_current_stream_deserializes_without_ss_offset` (legacy state.json backwards-compat)
- **SRT shift for seek — ffmpeg input-seek resets PTS** (Apr 15) — `ffmpeg -ss N -i input -vf subtitles=sub.srt` fast-seeks the input container, then resets output PTS to 0. The `subtitles` filter reads SRT cue times literally and matches them against the reset PTS, so a cue at `01:02:03` shows up 62 minutes into the seeked output when the user expects it at 62-N minutes. `transcode.rs::shift_srt` physically rewrites the SRT timestamps (`HH:MM:SS,mmm --> HH:MM:SS,mmm` lines) by `-N` seconds before ffmpeg reads them, clamping negative to `00:00:00,000`. Wrapped in `resolve_subtitle_path_for_seek` (four-way decision tree: None / no-seek-passthrough / zero-seek-passthrough / positive-seek-uses-shifted). Latent bug since `ab3bad7` fast-seek commit Apr 6, only surfaced Apr 15 with Smart Resume actually using seek values. Tests: `shift_srt_*`, `resolve_subtitle_*`, plus a realistic 30-min seek roundtrip
- **Duration-aware reaper grace period** (Apr 15) — old formula was a flat 45-minute post-ffmpeg cushion before `do_cleanup` fires, which killed 63-min TV episodes mid-playback (ffmpeg finishes transcoding at ~11 min wall, grace expires at 56 min, cleanup fires while user still at the 30-min mark). New formula via pure helper `compute_reaper_grace_secs(duration, ss_offset)`: `max((duration - ss_offset) + 10min cushion, 5min floor)`, or legacy 45-min default when duration is unknown. Handles 3-hour movies, respects seek offset for partial playback, floor protects short clips. Tests: `test_grace_covers_63_minute_episode`, `test_grace_covers_three_hour_movie`, `test_grace_respects_seek_offset`, `test_grace_floor_*`, `test_grace_unknown_duration_uses_legacy_default`
- **Ranker missed "H 265" / "H.265" / "H-265" space/dot-separated** (Apr 15) — playWEB release `The.Boys.S05E02.2160p.playWEB.H 265-GNOMiSSiON` ranked ABOVE the 1080p H.264 result because `is_hevc_from_title` only matched `h265`/`hevc`/`x265` as tight substrings. Rewrote with a codec-name normalization pass that collapses space/dot/dash/underscore between 'h' and '265'/'264', so all four variants produce the same token. `test_is_hevc_from_title_space_separated_variants` + integration test `test_ranking_playweb_space_separated_codecs_apr15` using the actual Apr 15 release titles. Generic lesson: torrent release-name token parsers must normalize separators before matching, because scene groups use all of them interchangeably
- **TV episode resume HWM collided on shared IMDb ID** (Apr 15 v3.0.0) — `resume_positions` was keyed by raw `imdb_id`, but for a TV show that's the SHOW id (`tt1190634` = The Boys as a whole). Every episode of the same show wrote to the same bucket: S05E02's last cast_health_monitor save wrote 3437s to `tt1190634`, and the next night's S05E03 auto-resume picked up that stale HWM and started the new episode at minute 57. The confusing part was that the `past_end` guard in cast_health_monitor's save block short-circuited the whole thing, so `save_position_smart`'s internal 92%-completion reset never fired on the last tick either — the HWM just sat there. Two-layer fix: (1) `state.rs::resume_position_key` now parses `SxxExx` out of the release title and appends `_sXXeYY` when present, so each episode gets its own bucket (movies keep raw imdb_id behavior — no S/E marker, no suffix); (2) cast_health_monitor's end-of-episode branch now explicitly calls `reset_position` before `do_cleanup`, belt-and-suspenders regardless of the save-block guard. Regression pins: `test_tv_episodes_do_not_collide_on_shared_imdb_id`, `test_movies_use_raw_imdb_id_key`, `test_extract_se_suffix_variants`. Same confusion surfaced earlier in the session as "is it really from the very beginning? I'm confused" — we just didn't trace the collision until later
- **`ss_offset` must be zero for non-transcoded plays** (Apr 15 v3.0.0) — `CurrentStream.ss_offset` was being set to `seek_to.unwrap_or(0.0)` unconditionally, but for non-transcoded H.264 plays spela calls `cast.seek(pos)` AFTER load and the Chromecast's `current_time` already reflects the seeked offset on its own timeline. Adding `ss_offset` to `current_time` DOUBLED it. The Apr 15 incident: second play of S05E03 with `--no-intro` → not transcoded → `cast.seek(3469)` → Chromecast reported current_time=3478 → `cast_health_monitor` computed `absolute = 3478 + 3469 = 6947` → `176% of duration (3947)` → end-of-episode detection → cleanup → stream killed 10 seconds after it started. Fix: `ss_offset: if is_transcoded { seek_to.unwrap_or(0.0) } else { 0.0 }`. Semantically `ss_offset` is "the `-ss N` passed to ffmpeg", not "the seek offset" in general
- **Stale Chromecast current_time after stream replacement** (Apr 15 v3.0.0) — when a cast session is replaced mid-flight (new play on same device), the NEW session can inherit the OLD session's `current_time` for a few polls before the receiver resets. If auto-resume sets a non-zero ss_offset, `absolute = stale_current_time + new_ss_offset` produces a physically-impossible advance (>2x realtime). `is_position_jump_suspicious(delta_wall, delta_abs)` rejects any tick where `delta_abs > 2*delta_wall + 60s`, and `cast_health_monitor` tracks `last_save_wall: Option<Instant>` alongside `last_saved_position` so each prospective save is sanity-checked against wall-clock reality. One-off glitch self-heals on next tick; persistent glitch keeps skipping until the monitor exits. Pure helper + 6 unit tests in `server.rs::tests`
- **Disk cap enforced by refusal, not by pruning** (Apr 15 v3.0.0) — `prune_disk` existed but had THREE compounding bugs that made it functionally a no-op in production: (1) `if !path.is_dir() { continue; }` guard silently skipped top-level files, so single-file torrent releases like `The.Boys.S05E01.FLUX.mkv` and `28.Years.Later.mkv` were immortal regardless of age. (2) `name.contains(active_title)` with an empty string matches everything, so `prune_disk(dir, "")` protected every entry and became a silent no-op. (3) `name.contains(active_title)` with a non-empty string was a literal substring match, so dot-separated torrent filenames never matched space-separated request titles → active-title protection failed in the test suite and probably in production too. All three fixed. Added `prune_to_fit(dir, active, target_mb)` which runs age-based prune first, then LRU-evicts oldest entries until under target — turns the cap from a refusal wall into a self-maintaining upper bound. Wired into `do_play` so new plays are NEVER blocked by stale cache. Incident trigger: Apr 15 evening, Ruby's S05E03 plays all bounced off the 10 GB cap because 13 GB of content from a week of testing (28 Years Later, S04E07/08, S05E01/02) was immortal. Six regression tests for prune_disk top-level files, empty active_title, active-title protection, LRU eviction, active-title LRU protection. Generic lesson: ANY age-based eviction policy needs an emergency pressure valve for the case where EVERYTHING is under the age threshold
- **`--seek N` (including 0) must reset the saved HWM** (Apr 15 v3.0.0) — `do_play`'s auto-resume logic only triggered when `req.seek_to.is_none()`, so `spela play 3 --seek 0` correctly skipped auto-resume BUT the existing saved HWM sat there untouched. Next implicit play (`spela play 3`, no `--seek`) picked up the old HWM and resumed at the stale position. The user's only clean-restart workaround was `spela clear tt1190634` + replay. Fix: explicit `req.seek_to.is_some()` branch clears the HWM via `reset_position` before auto-resume is even considered. Principle: explicit user actions override remembered state. The symptom was the same as the TV-episode collision ("confused about whether it's from the beginning") but the root cause was different — same session, different bug class, fix them together
- **Visible auto-resume notification** (Apr 15 v3.0.0) — `do_play` now returns `resumed_from: Option<f64>` in the JSON response, `Some(pos)` when auto-resume picked up a saved HWM, `None` otherwise. `print_human` formats it as `↩ Resuming at H:MM:SS (from saved position)`. Voice-assistant consumers can read the field directly. Prevents the "is it really from the very beginning? I'm confused" class of UX bug: when spela auto-resumes, you FIND OUT before you have to ask
